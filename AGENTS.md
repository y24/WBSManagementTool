# AGENTS.md - WBS Management Tool 開発ガイドライン

このドキュメントは、本プロジェクトの開発を継続・引き継ぐ開発者（およびAIエージェント）が遵守すべきルール、実装方針、および暗黙の仕様をまとめたものです。

---

## 1. プロジェクト概要
WBS（Work Breakdown Structure）管理ツール。プロジェクト、タスク、サブタスクの3階層で工程を管理し、ガントチャートで進捗を可視化するWebアプリケーションです。

- **Backend**: Python (FastAPI), SQLAlchemy, Pydantic, Alembic, SQLite
- **Frontend**: React (Vite, TypeScript), Tailwind CSS, Lucide React (Icons)

---

## 2. UI/UX 開発ルール（必須）

### ユーザーインターフェースのデザイン理念
- **プロフェッショナルな配色**: 派手すぎない、落ち着いたトーン（Muted colors）を優先してください。
  - ガントチャートの日付ヘッダなどは、情報を強調するための原色（赤など）の使用を避け、グレー背景と黒文字などのバランスで視認性を確保します。
  - マーカー（Milestones）などの補助的な指標も、プロダクト全体との調和を重視し、彩度を抑えた配色を維持します。
- **視覚的フィードバックの洗練**:
  - ガントチャートの進捗バー（Achievement bar）とレビュー部分の境界は、グラデーションやボカシ（Blur）を用いて滑らかに表現します。
  - ホバー時のプレビュー線などは、太すぎない控えめな設定を維持してください。

### モーダルウィンドウの挙動
全てのモーダル（詳細、削除、一括処理、エクスポート等）において以下の実装を徹底してください：
- **Escapeキー対応**: `keydown` イベントを検知し、Escapeキーでモーダルを閉じられるようにします。
- **未保存時の確認ダイアログ**: 編集モーダル等で「未保存の変更」がある状態で閉じようとした場合、必ず確認ダイアログを表示して誤操作を防いでください。

### ユーザ設定の永続化
- **UIオプションの保持**: 進捗率の表示/非表示、その他の表示オプションは `localStorage` に保存し、ブラウザをリロードしても設定が維持されるようにしてください。

## 3. 実装方針とデータ仕様

### バックエンド（FastAPI / SQLAlchemy）
- **デフォルトステータス**: 新規作成（ProjectCreate, TaskCreate, SubtaskCreate）時、`status_id` は原則として **1（"新規"）** をデフォルト値として設定してください。
- **論理削除**: データの削除は `is_deleted = True` による論理削除を基本としています。
- **DB初期化スクリプト**: `backend/setup_db.py` はマスタデータの定義とテーブルの初期化を担います。本番環境や開発環境の初期構築時に使用されます。

### フロントエンド（React）
- **データの階層性と影響範囲の可視化**: WBSのような階層構造データを削除する際は、ユーザーが意図しない範囲まで操作が及ぶのを防ぐため、削除対象の影響範囲（配下の全要素の再帰的な集計）を具体的に提示することを原則としています。
- **Excel連携**:
  - エクスポートされるExcelは、そのままインポート可能なフォーマットを維持してください。
  - カラム幅などはエクスポート機能側で適切に制御されており、ユーザーが使いやすい幅を維持する必要があります。

## 4. やってはいけないこと（制約事項）

- **UI/UX への過度な視覚的強調**: 日付やステータスなどの通常情報に対し、不要に注意を喚起する強烈な警告色を用いず、プロフェッショナルで落ち着いた（Muted）な配色の一貫性を維持してください。
- **制御の不備によるデータ消失と操作性の低下**: モーダル等のサブウィンドウには常に直感的な脱出路（Escapeキー）を提供し、同時に不注意による入力内容の消失を防ぐためのガード（未保存確認）を標準実装としてください。
- **基盤データの整合性破壊**: システムの動作が依存している予約済みデータ（`is_system_reserved=True` 等）を、設計意図を無視して不用意に変更・削除しないでください。

## 5. コーディング規約と設計原則

### 共通原則
- **型安全性の追求**: Backend（Pydantic）および Frontend（TypeScript）の両層で厳密な型定義を行い、エッジケースでの不具合を未然に防いでください。
- **DRY (Don't Repeat Yourself)**: 重複するロジック（日付計算、単位変換等）は共有のユーティリティ関数や共通コンポーネントに抽出し、一貫性を保ってください。
- **関心の分離 (Separation of Concerns)**:
    - **Backend**: DBモデル、リクエスト/レスポンス型（Schemas）、ビジネスロジック（CRUD）、APIルーティングの役割を明確に分離してください。
    - **Frontend**: UI（Components）、ページ構成（Pages）、API通信、ユーティリティ（Utils）の責務を分け、各ファイルが肥大化しないよう努めてください。

### バックエンド（Python/FastAPI）
- **非同期処理の適切な実装**: DBアクセス等のI/O待ちが発生する処理は `async/await` を活用し、スループットを維持してください。
- **エラーハンドリングの標準化**: 異常系では適切な HTTP ステータスコードを選択し、フロントエンドが扱いやすい統一された JSON 形式のエラーレスポンスを返してください。
- **可読性の高い命名**: 変数や関数名は、その役割を明示的に示す snake_case を使用し、ドキュメントとしても機能するコードを目指してください。

### フロントエンド（React/TypeScript）
- **コンポーネントの疎結合化**: 独自のビジネスロジックは可能な限り Custom Hooks に切り出し、コンポーネント自体は描画（UI）に専念させることでテストや再利用を容易にしてください。
- **命名規則の徹底**:
    - コンポーネントおよびファイル名: PascalCase (例: `TaskDetailModal.tsx`)
    - 定数、ユーティリティ、変数名: camelCase (例: `isDirty`, `fetchWbsData`)
- **パフォーマンスへの配慮**: 頻繁に再描画が発生するリスト表示や重い計算には `useMemo` や `useCallback` を適切に使い、ユーザー体験を損なわないようにしてください。

## 6. 開発者向けTips

- **API サーバーの起動**:
  ```powershell
  cd backend
  uvicorn app.main:app --reload
  ```
- **フロントエンドの起動**:
  ```powershell
  cd frontend
  npm run dev
  ```
- **初期セットアップ**: `server_setup.ps1` (PowerShell) が用意されています。
- **ドキュメント参照**: `docs/` ディレクトリに主要な設計ドキュメントが格納されています。実装前に必ず参照してください。
    - `01_requirements.md`: 全体要件定義
    - `02_ui_requirements.md`: UI/UX デザイン要件
    - `03_screen_item_list.md`: 画面項目定義・バリデーションルール
    - `04_db_design.md` & `db_definition.md`: データベース設計・テーブル定義
    - `05_api_list_design.md`: API エンドポイント一覧と仕様
    - `IIS_Setup_Guide.md`: Windows/IIS へのデプロイ手順
