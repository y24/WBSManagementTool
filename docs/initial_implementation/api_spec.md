# API 仕様書 (API Specification Document)

このドキュメントは、WBS Management Tool のバックエンド API 仕様をまとめたものです。各エンドポイントの役割、パラメータ、およびシステム内部で実行される自動計算ロジックや制約について詳しく説明します。

---

## 1. 共通仕様

- **通信形式**: JSON (REST API)
- **ベースURL**: `/api`
- **ステータス管理**:
  - `status_id`: マスタデータ `mst_status` に依存。
  - デフォルト値: 原則として `1`（"新規"）。
  - 特殊なステータス: `7`（"Removed"）は論理削除に近い扱いを受け、集計から除外される場合があります。

---

## 2. WBS データ集計系

### `GET /wbs`
WBS の全階層データ（プロジェクト > タスク > サブタスク）を取得します。

- **パラメータ**:
  - `project_ids`: フィルタリングするプロジェクトIDリスト（オプション）。
  - `include_done`: 完了済みのプロジェクトを含めるか (default: `false`)。
  - `include_removed`: 削除フラグが立っている、またはステータスが "Removed" のものを含めるか (default: `false`)。
  - `weeks`: ガントチャートの表示期間（週単位、今日以前の範囲計算に使用）。
- **返却値**: `WBSResponse` 型。プロジェクトのリストに加え、ガントチャートの描画範囲 (`gantt_range`) を含みます。
- **特記事項**:
  - 内部で `refresh_subtasks_actual_end_date` を呼び出し、進捗が 100% かつ実績終了日が未設定のサブタスクに当日をセットします。
  - ガントチャートの範囲は、全データの最小開始日〜最大終了日を基準に、前後1〜2週間のバッファを持って自動計算されます。

#### 自動計算ロジック (View Logic)
DBから取得したデータをレスポンスに変換する際、以下の項目が動的に計算されます。
1. **計画工数合計 (`planned_effort_total`)**: 配下のサブタスクの `planned_effort_days` の合計。
2. **実績工数合計 (`actual_effort_total`)**: 配下のサブタスクの `actual_effort_days` の合計。
3. **進捗率 (`progress_percent`)**:
   - 計画工数がある場合：配下のサブタスク（またはタスク）の進捗率を、計画工数で加重平均したもの。
   - 計画工数がない場合：配下の稼働率 (`workload_percent`) による加重平均。
   - いずれもない場合：単純平均。
4. **重複フラグ (`is_overlapping`)**: 配下項目の計画期間に重複がある場合に `true` となります。

## 3. CRUD 操作 (Project / Task / Subtask)

各エンティティの作成・更新・削除を行います。

### `POST /projects`, `/tasks`, `/subtasks`
新しい項目を作成します。
- **制約**: 作成時、`status_id` は指定がない限り `1`（"新規"）となります。
- **トリガー**: 作成後、WebSocket を通じてフロントエンドへ同期通知を送信します。

### `PATCH /projects/{id}`, `/tasks/{id}`, `/subtasks/{id}`
項目を更新します。
- **波及効果**:
  - サブタスクが更新されると、親タスクおよび親プロジェクトのステータス・日付が自動再計算されます。
  - タスクが更新されると、親プロジェクトのステータス・日付が自動再計算されます。

### `DELETE /projects/{id}`, `/tasks/{id}`, `/subtasks/{id}`
項目を論理削除します (`is_deleted = True`)。

### `POST /reorder` (各階層)
ドラッグ＆ドロップ等による並び順 (`sort_order`) を一括更新します。

## 4. 自動再計算ロジック (Recalculation)

バックエンドの `crud/recalc.py` に実装されている核心ロジックです。

### ステータスの自動遷移 (Bottom-up)
下位階層の状態に基づいて、上位階層のステータスが自動的に決定されます。

- **タスクステータスの決定基準**:
  1. 配下に一つでも **"Blocked"**（保留/ブロック）があれば、タスクは **"Blocked"**。
  2. 配下がすべて **"Done"**（完了）であれば、タスクは **"Done"**。
  3. 配下がすべて **"New"**（新規）であれば、タスクは **"New"**。
  4. 上記以外は **"In Progress"**（進行中）。

- **プロジェクトステータスの決定基準**:
  - 基本的にタスクと同様ですが、タスクのステータスをベースに判断されます。

### 日付の自動集計
`is_auto_planned_date` または `is_auto_actual_date` が `true` の場合、親項目の期間は配下項目の最小開始日〜最大終了日に自動設定されます。手動で日付を変更しても、配下が存在する場合はこのロジックが優先されます。

## 5. 一括処理 (Batch Operations)

### `POST /items/duplicate` (複製)
指定したプロジェクト、タスク、サブタスクを複製します。
- **仕様**: 親子が同時に選択されている場合、重複して複製されないよう最適化されています。複製された項目は `sort_order` が末尾に設定されます。

### `POST /items/clear-actuals` (実績クリア)
実績に関するデータ（実績開始日、実績終了日、進捗率、実績工数）を一括でリセットします。
- **仕様**: 指定した項目の配下にある全子孫要素も再帰的にクリアされます。クリア後、上位階層の進捗率やステータスが再計算されます。

### `POST /items/shift-dates` (日付移動)
選択した項目の期間を、営業日を考慮して一括でスライドさせます。
- **パラメータ**: `new_base_date` (新しい基準日)
- **仕様**: 
  1. 選択された全項目の中から最小の開始日（基準日）を特定。
  2. `new_base_date` への差分を **営業日単位**（マスタ設定の休日を考慮）で計算。
  3. 全対象項目の全日付（計画・実績・レビュー）に対し、計算した営業日オフセットを適用。

## 6. インポート・エクスポート

### `GET /import/template`
インポート用の Excel テンプレートを生成します。

### `POST /import/preview`
アップロードされた Excel を解析し、取り込み内容のプレビューとバリデーションを返します。
- **階層判定**: 行の「レベル」情報に基づき、Project(0) > Task(1) > Subtask(2) と判定します。
- **バリデーション**: 日付の整合性、マスタ（担当者、ステータス、種別）の存在チェックなど。

### `POST /import/execute`
プレビュー済みのデータをもとに、一括でDBへ登録します。

### `POST /wbs/export`
現在のフィルタリング内容に基づいた Excel ファイルを出力します。出力されるファイルは、そのままインポート可能なフォーマットを維持します。

---

## 7. その他

### `GET /dashboard`
プロジェクトの全体状況を可視化するための統計データを取得します。
- **KPI項目**: 進行中プロジェクト数、着手遅延数、期限超過サブタスク数など。
- **乖離率**: `(実績工数 - 計画工数) / 計画工数` で算出。

### `GET /initial-data`
フロントエンドの起動時に必要なマスタデータ（ステータス、担当者、休日、システム設定等）を一括取得します。

### `POST /masters/holidays/sync`
外部 API（holidays-jp）から日本の祝日情報を取得し、ローカルの休日マスタを更新します。
